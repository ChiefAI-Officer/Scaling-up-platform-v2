## Item 1 — Pager merge (fewer clicks)

1. **Core:** In `src/src/components/assessments/section-pager.tsx`, collapse the `view: 'intro' | 'questions'` two-step into ONE page per section: render the intro content (accent rail, section title, the styled "What this section covers" callout `su-intro-covers`) ABOVE the question list on the same page. Remove the `view` state, `handleForwardFromIntro()`, and the second "Next". One "Next" advances section→section; `handleNext()` keeps its required-field validation (filter `q.isRequired && !isAnswered` → `invalidKeys` → focus-first-invalid) BEFORE advancing/submitting.
2. **Question-less pages stay:** `buildSectionPages()` (`src/src/lib/assessments/section-pages.ts`) still maps EVERY section to a page (incl. LVA Welcome/Completion, which have a description but 0 questions). Those render intro-content-only + Next — no validation to run. `pageHasIntro()` becomes moot but removing question-less pages is explicitly OUT (a separate follow-up pending Jeff).
3. **Heading is now single per page** — the merged page renders ONE `<h2 className="survey-section-title" tabIndex={-1}>` (drop the duplicate `su-intro-title` heading id `su-intro-heading`); `focusHeading()` (RAF `.focus()` on the tabindex=-1 heading) now targets it on every advance/back.
4. **Edge/unhappy:** empty `description` → no callout box, questions render flush (today's `?.trim()` guard preserved); a section with 0 required + 0 description still gives a valid single-Next page; first page Back is wired to `onExit` (not dead); last page Next → `attemptSubmit`. Malformed/missing `page.questions` defends to `[]`.
5. **Tests:** rewrite `src/src/__tests__/assessments/section-pager.test.tsx` (~15) for one-page-per-section: assert intro+questions co-render, single Next advances, validation still blocks + focuses first invalid, heading focus on advance/back, question-less page advances without validation.

## Item 3 — Invite-email body + subject defaults

8. **Core — defaults live in a client-safe constants module (NOT the server renderer):** create `src/src/lib/assessments/invitation-defaults.ts` — a tiny, side-effect-free module exporting `DEFAULT_INVITATION_BODY` (markdown; tokens `{{respondentFirstName}}` / `{{organizationName}}` / `{{templateName}}`) and a static `DEFAULT_INVITATION_SUBJECT`. Both the server renderer (`invitation-email.ts` / `notifications.ts`) AND the client `CampaignWizard` import from THIS module. Do NOT export them from `invitation-email.ts` — that module pulls `Buffer`/`sanitize-html`/the CID logo into the bundle, and importing it from the client wizard would drag server-only deps into the browser. Add a guard that `CI=true npx next build --turbopack` stays clean with the wizard importing the new module. `{{templateName}}` already resolves in `buildTokenValues`.
9. **Single chokepoint, normalize BEFORE the branded/legacy split:** in `sendAssessmentInvitationEmail` (`src/src/services/notifications.ts` ~L1093), compute `effectiveSubject`/`effectiveBodyMarkdown` (apply defaults when blank) ONCE at the top — BEFORE the `branded = ASSESSMENT_INVITE_BRANDED !== "0"` split (~L1114) — and pass them into BOTH the branded path (subject ~L1137, markdown body ~L1165) AND the legacy renderer (`sendAssessmentInvitationEmailLegacy` ~L1189). Custom-HTML (`fullHtml`, ~L1158) still bypasses the *body* default (its body comes from HTML), but the *subject* default still applies (subject never comes from HTML). Empty-aware: treat `null`/`undefined`/whitespace-only as blank (`??` does NOT catch `""` — test `.trim()`).
9a. **Harden the legacy subject (security, in-scope because item 9 touches it):** the legacy renderer's `substitute()` (~L1209) builds the subject WITHOUT `SUBJECT_ALLOW`/`renderSubject`, so a token value carrying `{{invitationUrl}}`'s `#t=<token>` or CR/LF could leak the invite credential into the SMTP subject header/telemetry or attempt header injection. Route the legacy subject through `renderSubject`/`SUBJECT_ALLOW` AFTER default selection; extend the legacy body `substitute()` to resolve `{{organizationName}}`/`{{templateName}}`. Add legacy-mode tests proving `#t=` and CR/LF are stripped from the subject.
10. **Clobber worry:** substitute ONLY when the authored field is blank; any admin-authored body/subject passes through unchanged. Reminder/resend paths route through the same chokepoint, so the default reaches every send path consistently.
11. **Wizard preview, NOT value-prefill:** show `DEFAULT_INVITATION_BODY`/`DEFAULT_INVITATION_SUBJECT` as the textarea/input **placeholder** (or a read-only "default preview" + a "Customize" affordance) in `src/src/components/assessments/CampaignWizard.tsx` (~L1896) — do NOT write the default into `state.invitationBodyMarkdown`. Reason: a value-prefill that the coach never edits would persist as a non-null per-campaign override, freezing that campaign to the prefilled text and cutting it off from the chokepoint default (and future default updates). Unedited → field stays empty → save path (~L491) submits `undefined` → the chokepoint applies the live default. (Dirty/diff tracking is an acceptable alternative to placeholder.)
12. **Edge + token surfacing:** add `{{organizationName}}` and `{{templateName}}` to the wizard's supported-token helper list (currently omits them) so coaches can see/edit every token the default uses; missing `templateName`/`organizationName` → existing neutral token fallbacks (no literal `{{...}}` leaks); subject stays the token-allowlisted (`SUBJECT_ALLOW`) field and is previewed via placeholder, consistent with item 11.

## Item 4 — QSP v2 closing-section description (DRAFT re-seed)

13. **Core:** add a neutral closing description to the `P5_closing` section in `src/prisma/seed-qsp-v2-assessment.ts` (audit `wseh8bws4` confirmed it is the ONLY section lacking a description across all assessments). Re-seed via `ensureTemplateVersionContent` → content-hash diff appends a NEW DRAFT version vN+1.
14. **Data integrity / never-mutate-published:** the helper is append-only and content-hash idempotent — it never edits a published row and no-ops if latest content matches. Publish is a SEPARATE admin step (out of scope here; DRAFT only). Adding a `description` changes neither `stableKey` nor question `sortOrder`, so `assertSeedContentIntegrity` (global stableKey + global sortOrder uniqueness) still passes — assert this in a test before re-seeding.
15. **Fail-closed IN CODE (CORRECTION — parameterize, don't rely on a manual preflight):** `runSeed()` currently hardcodes `{ forceSupersedeDraft: true }` (`seed-qsp-v2-assessment.ts:489`, same pattern as all four seeds) — so it SILENTLY supersedes a divergent unpublished DRAFT (e.g. live reviewer edits). A manual preflight is not enough (a skipped check = silent clobber). **The fix:** parameterize `runSeed(client, { force = false })` — default fail-closed; the Wave G invocation passes nothing, so a divergent latest DRAFT ABORTS by default and superseding requires an explicit, audited `{ force: true }` override. Add QSP v2 runner tests: (i) divergent latest DRAFT aborts with no `force`; (ii) `force: true` appends a new DRAFT. A pre-run latest-version inspection stays as belt-and-suspenders. Scope: only QSP v2's `runSeed` changes (the other three seeds' hardcoded force is out of scope; follow-up to harmonize). Re-seed is a guarded, reversible, manually-run step — never auto-run on deploy.
