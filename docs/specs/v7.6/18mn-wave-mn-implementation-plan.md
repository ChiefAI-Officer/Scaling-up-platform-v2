## Wave M — custom slides (coach, per-campaign)
3. **Migration (M):** one additive nullable `AssessmentCampaign.customSlides Json?`. Run `check-migration-safety.mjs`; confirm additive (no NOT NULL, no drop, no backfill). `prisma migrate` name `add_campaign_custom_slides`. Existing campaigns read `null` ⇒ zero behavior change.
4. **`lib/assessments/custom-slides.ts` (new):** `CustomSlide` type + Zod schema (`id` cuid, optional `title`, `html`, `position` = `start|before-section:<stableKey>|end`, `sortOrder`); caps ≤10 slides, ≤20 KB post-sanitization/slide, title length cap; `mergeCustomSlides(pages, slides)` pure fn inserts at start / before anchored section / end-after-Other, ordered by `sortOrder`; unknown anchor stableKey at RENDER ⇒ drop+log (fail-safe, legacy-corruption only — see §8 for save-time validation); empty slide (no title AND empty sanitized html) ⇒ rejected at save / skipped at render.
5. **`sanitizeSlideHtml` (new, stricter than `sanitizeCustomHtml`) — SERVER-only** (R1-High-2): v1 **drops ALL `style` attributes and `<style>` blocks entirely** (no coach CSS — inline-`url()`/`@import` validation is bypassable via CSS escapes/nesting), drops `<iframe>`; `<img>` `src` allows `https` + `data:` but **NOT `data:image/svg+xml`**; drop/validate `srcset`; keep links `https`/`mailto`/`tel` + text formatting. Returns sanitized html + strip-warnings. TEST fixtures (must all be neutralized): `<script>`, `on*` handlers, `<style>{…}`, escaped/nested `url()`, `@import`, protocol-relative `//evil`, `srcset`, `data:image/svg+xml`, `javascript:`; bold/list/https-img survive. **External-image privacy (R2-Low-1):** external `https` images are a coach-controlled tracking-pixel vector (observe survey load + participant IP/UA). v1 mitigates with `referrerpolicy="no-referrer"` forced on every slide `<img>` + a `no-referrer` meta on the slide-rendering survey surfaces, and a documented privacy tradeoff (coach is semi-trusted, like email images); a metadata/test check ensures future PUBLIC-slide plumbing never leaks slide URLs. Image proxying/allowlist is a deliberate later add.
6. **Pager render (`section-pager.tsx` + `section-pages.ts`) — client renderer stays DUMB** (R1-Med-2): all `customSlides` are sanitized **server-side in the survey loaders** (§8) and serialized as a typed `SafeSlide` payload; the client pager only inserts the already-safe html via the raw-HTML prop (no sanitizer import in the browser bundle). Extend the page model to a discriminated union `{kind:"section"…}|{kind:"slide",id,title,safeHtml}`. Slide page renders a branded interstitial with **no shell header / no section counter** (match the existing `phaseTile` interstitial, `:47`); `totalSections`/`currentSection` count `kind==="section"` only; trailing `end` slide forward button = Submit; Back at index 0 ⇒ `onExit`; focus the slide title on advance (a11y). Gate is a no-op on zero-question pages (confirmed) — assert in tests. **Exhaustive `kind` handling (R3-Low-2):** every existing section-only path must guard for slide pages — the SU-Full phase-tile trigger (`currentPage.questions.some(...)`), `answeredCount`/`total` (`pages.flatMap(p=>p.questions)` is safe since slides carry `questions:[]`, but assert it), and `isLast`/Back/Submit; tests for a slide adjacent to the SU-Full phase tile, answered counts with slides present, and the required-answer gate on a slide.
7. **`CustomSlidesPanel.tsx` (new):** list (title + html textarea + position picker [Start / Before §▾ from the campaign's pinned-version sections / End] + reorder ↑↓ + remove), `+ Add slide`, inline strip-warnings, sandboxed-iframe live preview. Mounted in `CampaignWizard` (new optional step before Review) AND CampaignDetail (`/portal/assessments/[id]`).
8. **API (M):** `POST /api/assessment-campaigns` accepts `customSlides` in the create payload. **Create-path atomicity (R3-Med-2):** the campaign row + `customSlides` + participant attach + `tx.auditLog.create` all commit in ONE `db.$transaction`, and **immediate-send is emitted only AFTER that transaction commits** — so participant-facing slide HTML is never reachable without its audit row; a route test proves an audit-insert failure aborts the whole create. (Today's create writes the campaign first + swallowing `logAudit` after — do not inherit that.) The wizard posts an **`expectedVersionId`** (+ content hash); the server **validates every slide anchor against the resolved pinned version on save** and rejects stale/unknown anchors with **400** (R1-Med-1) — render-time drop+log is reserved for legacy corruption only. `PATCH /api/assessment-campaigns/[id]` accepts `customSlides`: flag-gated write (§2); `canManageCampaign(…,"write")`; reject on `CLOSED` (409), allow `DRAFT`+`ACTIVE`; **required value-compare CAS via `expectedCustomSlides` sentinel ⇒ 409 on stale** (R1-Med-5, mirrors Wave B); server `sanitizeSlideHtml` on save + caps. **Atomic mutation+audit (R2-Med-1):** the CAS update and `tx.auditLog.create` run in ONE `db.$transaction`, rolling back the slide write if the audit insert fails (the shared `logAudit` helper swallows post-update failures, so do not rely on it here); audit records prior-value metadata + html hashes/lengths (not necessarily full bodies). Survey loaders (`/org-survey/[alias]/me` server route, public quiz loader, report-preview) **re-sanitize server-side** and emit the `SafeSlide` payload.
9. **M edge/integrity:** mid-campaign edit only affects new survey loads (read-at-load); in-flight respondents keep their loaded copy + answers (autosave is answers-only, `sectionIndex` not persisted ⇒ resume safe even if slide count changed). Concurrent coach edits use the required CAS sentinel (§8) ⇒ 409, never silent clobber. Public campaigns: renderer supports slides but v1 has no authoring path ⇒ public payload omits.

## Wave N — per-respondent longitudinal (coach, scored-only)
10. **No migration.** Reads frozen `AssessmentSubmission.result` + `OrgRespondent`. Add `RESPONDENT_LONGITUDINAL_VIEW` to the `AuditAction` union (free-form string, no migration).
11. **`lib/assessments/respondent-longitudinal.ts` (new):** `getRespondentLongitudinal(db, actor, organizationId, respondentId, templateId)` → discriminated outcome `forbidden|notApplicable|empty|ok`. **Authz (two gates, both current — R2-High-2):** `canAccessOrganization` AND `canAccessTemplate(actor, templateId)` ⇒ else `forbidden`/404 (matches the sibling trends page, which already gates per-template access; a coach who owns the org but LOST template access must not read that template's named PII). **Entry-respondent binding (R2-High-1):** after authz, load the path `OrgRespondent` by `{ id: respondentId, organizationId, deletedAt: null }` and **404 on mismatch** — never let a stale/cross-org/soft-deleted id seed the union; then restrict every email-matched row to the SAME org + live rows. Scope gate: `reportConfigFor(alias).reportType !== "scored"` ⇒ `notApplicable:"qualitative-template"` (no load). **Match by `normalizedEmail`-within-org** (union all live same-org OrgRespondent rows sharing it; indexed) falling back to `respondentId` when no email. **One point per campaign, enforced (R1-High-3):** the DB unique is per `(campaignId, respondentId)`, not per email-person, so after resolving matched respondent IDs, if a single campaign has >1 matched submission, collapse deterministically (tie-break order, §12) and surface a `degraded`/ambiguity note — never plot two same-campaign points. Include Esperto-imported (CLOSED, back-dated) submissions.
12. **Comparison math:** order by a **deterministic stable key (R2-Med-2):** `submittedAt asc, campaign.openAt asc, campaignId asc, submissionId asc` (Esperto backfills + duplicate rows can share a `submittedAt`); this same order drives both delta computation and the same-campaign collapse (§11), with equal-timestamp tests. Project per submission `overallAverage` + `scaleUpScore?`/`perDomain?` (SU-Full, provisional label) + `tier?` + `perSection[].averagePoints`. **New tested version helper (R1-Med-3):** the existing `trends.ts` partition hard-EXCLUDES non-latest versions and has no same-version delta helper — build a fresh helper that PRESERVES all versions and defines the delta as **vs the previous same-`versionId` submission chronologically** (not merely the immediately-previous chronological one). Cross-version (no prior same-version) ⇒ value shown, delta `—` + "different version" badge; all-different-versions ⇒ values, zero deltas + note. Never re-score; `degraded` on malformed result (skip that column + note). **Capacity bounds (R3-Med-3):** the email-union can fan out on a large/duplicated org, so the query is bounded — cap matched respondent rows, total submissions, and rendered columns (e.g. most-recent N campaigns) with an explicit "showing latest N" degraded state rather than rendering an unbounded wide table of frozen result JSON. Emit PII-free latency/row-count metrics. A per-actor rate-limiter stays OFF by default (trends parity, GM-2) but the bounded query shape + `canAccessOrganization`+`canAccessTemplate` gates are the documented, tested DoS rationale.
13. **Route + UI (N):** portal page `/portal/assessments/respondents/[respondentId]/longitudinal?templateId=&organizationId=` + API. **Page is COACH-ONLY via `requireCoach`** (R1-Low-1, consistent with the sibling trends page); the `canAccessOrganization` admin-bypass is authz-fn/API defense-in-depth only, NOT an admin page entry (admins use admin surfaces). Page + API enforce BOTH `canAccessOrganization` AND current `canAccessTemplate` (§11, R2-High-2). `Cache-Control: no-store, private` + `no-referrer` on **both the page AND the API response** (R3-Low-1, route-tested); lightweight `RESPONDENT_LONGITUDINAL_VIEW` audit = **respondent id + matched-respondent-count + submission-count (+ optional fingerprint hash); NO raw emails** (R1-Med-6). NO rate-limiter / NO report-access-gate (matches sibling trends posture; `canAccessOrganization` 404s non-owners). `RespondentLongitudinalView` component + scoped CSS: header (name·company·assessment·"N assessments, first–last") → overall trend line → per-section/domain table (cols = dated campaigns, cells = avg + delta) + tier-movement row (only where a real tier exists; SU-Full has none per ADR-0015) → version note → states.
14. **Entry points:** per-respondent report ("View across campaigns") + CampaignDetail respondent list; both `prefetch={false}`. **Eligibility = the actor currently passes `canAccessTemplate` AND the person has ≥2 scored submissions for this template** (R1-Med-4 / R2-High-2, NOT ≥2 same-version — so the all-different-versions value-table still surfaces); the direct route, link visibility, and tests all align to this exact rule.
15. **N edge/unhappy:** 1 submission ⇒ render it + "need ≥2 to compare"; qualitative alias ⇒ not-applicable card; same-person-2-rows-no-shared-email ⇒ won't link (documented limitation + future merge tool); same-person-2-rows-same-campaign ⇒ collapsed + degraded note (§11); public takers excluded; deleted org/campaign ⇒ `canAccessOrganization`/live-only filters apply.
