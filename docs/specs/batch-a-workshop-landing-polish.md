## Item 3 — Timezone abbreviation on every event-time display

8. **Helper** `formatTimeWithZone(eventTime, eventDate, timezone)` exported from `lib/utils.ts`. **Parser-independent** (eventTime is free-form: `"16:00 - 18:00"`, `"9:00 AM"`, `"TBD"`, null): it does NOT parse `eventTime` and does NOT use `resolveEventStartMoment`/`setWallClockInTimezone` — the latter re-derives the calendar date *in the target zone* from the input (`formatToParts(date)`), so a midnight-UTC `eventDate` shifts Americas zones to the **previous** local day → wrong DST label (Codex r3 HIGH-1, verified at `resolve-event-start-moment.ts:63-86`). Instead: take the event's **UTC** calendar date (`eventDate.getUTC{FullYear,Month,Date}()` — `eventDate` is stored as midnight-UTC of the event day, so its UTC components ARE the event day), build a **noon-UTC** instant on that date (`new Date(Date.UTC(y, m, d, 12,0,0))` — maps to 02:00–08:00 local across all 9 Americas/Pacific zones, same calendar date, safely past the 2 AM DST transition), then `Intl.DateTimeFormat("en-US",{timeZone, timeZoneName:"short"}).formatToParts(...)` → `"CDT"`. Append to the passthrough `eventTime` → `"9:00 AM CDT"`. DST-correct for any real workshop regardless of `eventTime` format.
9. `eventDate` arg is the **stored date-only value** (`workshop.eventDate`, midnight UTC), NOT a pre-resolved instant. **Must never throw** (runs in SSR render): wrap in try/catch → on empty/`"TBD"`/null time, or invalid IANA `timezone` (RangeError), return the time **unchanged, no zone**.
10. **Zone coverage:** the 9 form-list zones map via `Intl` to `EST/EDT, CST/CDT, MST/MDT, PST/PDT, MST` (Phoenix no-DST), `AKST/AKDT, HST, EST/EDT` (Toronto), `PST/PDT` (Vancouver). No hand-maintained table.
