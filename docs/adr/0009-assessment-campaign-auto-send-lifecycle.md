# Creating an assessment campaign sends its invitations automatically; "Immediately" sends on create and future-dated campaigns auto-send when they open via a cron, with `openAt` unified as both the open-time and the send-time

> **Updated 2026-06-15 (post-`/claudex:plan`):** two corrections to the original decision below. (1) **No new `SCHEDULED` enum value** — "scheduled" is a DERIVED state (`status=DRAFT + inviteTiming=ON_OPEN + invitesSentAt IS NULL + openAt>now`); the cron flips `DRAFT→ACTIVE` at open. A persisted new enum value would break old code on rollback (R1-H3/R3-H1). (2) **Rollback is NOT a clean promote-previous** — rolled-back code reads `DRAFT+ON_OPEN` rows as ordinary drafts (auto-send + cron vanish), so a rollback requires a runbook (flag-off → pause cron → handle unsent ON_OPEN rows). Default-OFF feature flags + an `ASSESSMENT_SENDS_PAUSED` kill switch gate the lifecycle. See `docs/specs/v7.6/17d` §/claudex:plan hardening. The text below is the original rationale; where it says "SCHEDULED status" read "derived scheduled state", and where it claims clean promote-previous, see correction (2).

Creating a campaign no longer leaves it as an idle `DRAFT` that a coach must later activate and manually send. A **timing radio** (default **"Immediately"**) governs the campaign lifecycle: "Immediately" sets `openAt = now`, activates the campaign, and sends invitations to all participants on create; **"When the campaign opens"** sets a future `openAt`, parks the campaign in a new `SCHEDULED` status, and an Inngest **cron** (`*/3 * * * *`) sends the invitations when `openAt` arrives — **no manual trigger**. `openAt` is the single source of truth for both *when the survey is takeable* (pre-existing gate) and *when invitations fire*.

## Context

Jeff (June-9 feedback, `From Jeff/gabriel-feedback.docx`) asked for two paired things: *"Add a radio button for timing that defaults to 'Immediately'"* (#2) and *"Invitations should send immediately upon campaign creation. For future-dated campaigns, invitations should auto-send when the campaign opens on that date — no manual trigger required"* (#3). Today the flow is the opposite: `POST /api/assessment-campaigns` creates a `DRAFT`, the coach activates it, then explicitly calls `…/invite` to send. There is no scheduled-send mechanism anywhere in the codebase (the only assessment cron is the `quick-assessment-lead-email` outbox drain at `*/3 * * * *`; workflow steps use `step.sleepUntil`).

Grilled via `/grill-with-docs` (2026-06-15). A key discovery during the grill reshaped the design: **`openAt` already gates survey accessibility** — the `exchange`, `me`, and both `submit` routes reject access with "This survey hasn't opened yet" when `now < openAt`. A naive "send immediately" against an independent future `openAt` would therefore email everyone *now* but slam the door until `openAt`, stranding recipients on a "not opened yet" page — a state Jeff never asked for. This forced the unification of send-time with open-time. Design doc: `docs/specs/v7.6/17d-wave-d-campaign-setup-design.md` (Spec 17 Wave D).

## Considered options

- **Keep the manual flow, just add a radio** — rejected: Jeff explicitly wants "no manual trigger required." Retaining a manual gate contradicts the ask and leaves the future-dated case unsolved.
- **Decouple invite-timing from `openAt`** (send now even if `openAt` is future) — rejected: produces the "invited but the survey is closed" trap (recipients hit "not opened yet"). Adds a confusing dead state for no stated benefit.
- **Unify `openAt` as open-time *and* send-time (chosen)** — "Immediately" means `openAt = now`; "When it opens" means send at `openAt`. Invitations and survey-availability always coincide. One mental model, no dead state, faithful to "send when the campaign opens."
- **Schedule future sends with `step.sleepUntil` (a per-campaign sleeping timer)** — rejected: `openAt` is editable post-create (CampaignDetail has an inline editor), so a sleeping run would hold a stale time and require explicit cancel/reschedule; long sleeps span redeploys. A **re-scanning cron** self-corrects on every edit, survives missed runs/redeploys, and reuses the existing outbox + retry idiom. Chosen.
- **A confirm modal before the now-destructive "send now"** — rejected (grill Q1): coaches habituate to clicking through modals. Instead the wizard's existing Step-4 Review screen is the gate, and the final button is **consequence-labeled** ("Create & send N invitations now" vs "Schedule for &lt;date&gt;") so the action's effect is unmistakable.

## Consequences

- **Creating a campaign is now a sending action.** A future reader seeing `POST /assessment-campaigns` enqueue emails — or a cron that flips `SCHEDULED → ACTIVE` and blasts invitations — should not treat it as a bug or a security issue. It is deliberate, per Jeff's #2/#3. The **human review gate is the wizard's Review step + the consequence-labeled button**; there is intentionally no manual post-create send step and no DRAFT pre-state for "Immediately."
- **New `SCHEDULED` status** on `AssessmentCampaignStatus` (additive enum value) for the create→open window of future-dated campaigns.
- **The cron is the scheduler.** `*/3 * * * *`; predicate `status = SCHEDULED AND inviteTiming = ON_OPEN AND openAt ≤ now AND invitesSentAt IS NULL AND deletedAt IS NULL`. Worst-case lag past `openAt` is ~3 minutes — acceptable because campaigns open on a *date*. **Legacy campaigns are never swept** (they default `inviteTiming = IMMEDIATELY` and are already ACTIVE/sent).
- **Idempotency is load-bearing, not deferred.** The cron claims a campaign with an atomic `updateMany` gated on `invitesSentAt IS NULL` (single-writer CAS); per-recipient outbox rows carry an `idempotencyKey`. An Inngest replay or overlapping cron tick can never double-send. This closes the punch-list's previously-deferred "per-recipient pre-send idempotency."
- **`openAt` edits reschedule, never resend.** On a `SCHEDULED` (unsent) campaign, editing `openAt` reschedules (the cron re-scans; set to the past → next tick sends). Once `invitesSentAt` is set, the `openAt` editor is **locked** — no edit can resend, and the survey can't be re-closed on already-invited people.
- **Additive migration only.** `inviteTiming` enum (`IMMEDIATELY` default) + `invitesSentAt DateTime?` + the `SCHEDULED` enum value. All nullable/defaulted, so promoting the previous Vercel deployment remains a valid rollback (old code ignores the new columns) — but note this is the first Wave-D wave that requires a migration at all (Waves A–C were zero-migration).
- **Scope:** this ADR covers the invited-campaign auto-send lifecycle. The public quiz path is unchanged. The per-campaign custom HTML email (#20) and the results/notify toggles (#15/#16) are separate Wave-D items documented in 17d. Related: ADR-0006 (imported campaigns are closed campaigns) interacts with status semantics — imported campaigns are created CLOSED and are never SCHEDULED.
