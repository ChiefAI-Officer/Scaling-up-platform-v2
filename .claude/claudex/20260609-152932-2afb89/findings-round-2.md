# Round 2 findings

## High
- Public result-email routing can still be abused as a spam relay and SU-team notification flood because only the coach recipient is allowlisted while arbitrary taker emails and the fixed SU inbox sit behind a public, fail-open/IP-only rate limit. (Add bot controls, fail-closed distributed rate limits, per-recipient/per-campaign quotas, bounce suppression, and send coach mail only after an ACTIVE coach match.)
- Public submit plus result email has no idempotency or durable outbox, so crashes, timeouts, browser retries, or SMTP partial failure can create duplicate submissions, duplicate emails, or persisted submissions whose lead/result emails are lost forever. (Persist a client attempt id or submission fingerprint under a unique constraint and enqueue recipient-specific outbox jobs in the same transaction as the submission.)
- The planned Five Dysfunctions team aggregate would inherit a real authorization gap because the trends page/API gates only organization ownership, not template access or per-campaign read rights, so direct URL/API calls can expose licensed or sensitive results for templates/campaigns the actor should not read. (Gate aggregate reads with canAccessTemplate plus canManageCampaign("read") for every included campaign, or create a dedicated team-report route with explicit permissions and URL-tampering tests.)
- The no-prod-scripts safety claim is incomplete because the current campaign creation path always creates DRAFT/INVITED campaigns and the admin template UI disables PUBLIC mode, so launching the public Quick Assessment still requires direct DB mutation or an ad hoc path. (Build an audited admin-only PUBLIC campaign create/publish flow with alias, accessMode, lifecycle-window, publicConfig, and rollback validation.)
- The "no scoring change" Five Dysfunctions plan cannot represent the stated High/Medium/Low decimal boundaries exactly because current per-domain tier validation requires touching ranges and the resolver uses inclusive bounds, so 3.25-style boundary scores either fail publish validation or fall into the wrong band. (Add explicit half-open/range-comparator semantics or a licensed-toolset banding adapter, then test 3.24/3.25/3.74/3.75 boundaries before publishing.)

## Medium
- Admin draft template edits and publish are last-write-wins with no contentHash/CAS guard, so two admins can overwrite each other's question/scoring changes or publish stale half-reviewed content. (Require an expected contentHash or updatedAt on PATCH and publish, return 409 on mismatch, and include the hash in the publish audit event.)
- Returning ScoreResult from the public submit response without no-store headers can cache PII and assessment results in browsers, shared proxies, or edge caches despite the "no public results endpoint" design. (Mirror the invited submit route with Cache-Control: no-store on every public submit response and keep results out of URLs/history.)
- Public submissions and lead-routing decisions are not auditable today, leaving no reliable trail for spam investigations, coach attribution disputes, or PII-disclosure reviews. (Audit public submission creation, result return, coach match/suppression, recipient list, idempotency key, IP, user agent, and delivery telemetry linked to submissionId.)
- The per-domain team aggregate is underspecified for missing answers, duplicate public attempts, and weighting, so implementation may silently mix pooled-answer averages, mean-of-respondent means, zero-filled null domains, or mismatched n-values. (Define the canonical formula, require all Five Dysfunctions items for licensed runs, dedupe by invited participant where applicable, carry n per domain, and distinguish null/no-data from a real low score.)
- The plan's privacy/consent copy conflicts with the new routing behavior because the existing public UI says responses are shared only with the coach/facilitator while the plan sends results to Scaling Up and possibly a matched coach. (Update pre-submit copy, privacy disclosure, retention policy, and email-recipient consent before collecting public lead data.)

## Low
- referringCoachEmail is accepted and stored without canonical normalization in the current public route, so case/format drift can cause missed coach matches or fragmented attribution. (Trim/lowercase before lookup and storage, and match only against a unique ACTIVE coach email.)
