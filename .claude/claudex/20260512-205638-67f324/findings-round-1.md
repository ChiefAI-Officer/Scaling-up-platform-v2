# Round 1 findings

## High
- Wave 11-G targets the wrong layer: `execute-workflow.ts` and `trigger-workflow-step.ts` only pass `surveyTemplateId`, while the active-only rejection is in `getOrCreateSurveyLink` in `survey-automation.ts`, so dropping a clause in the named executor files will not fix inactive pinned templates. (Change the shared helper contract so explicit `templateId` lookup omits `isActive` while category auto-match keeps `isActive: true`; add helper-level and scheduled/Trigger Now integration tests.)
- The directions-button plan assumes a conditional email-template renderer that does not exist: `composeRegistrationConfirmationEmail` only interpolates simple `{{token}}` values and has no workshop format, venue, `showDirections`, or block syntax support, so `{{#if showDirections}}` would either render literally or only fix the hardcoded fallback, not the admin override body. (Extend the composer context and implement a real supported conditional/block-stripping mechanism or a generated directions token; update the admin default/production template path and test both override-enabled and fallback modes.)

## Medium
- Hiding directions for VIRTUAL only will still show a broken/meaningless directions button for HYBRID or IN_PERSON workshops with no mappable physical venue, which the API/schema permit even if the wizard tries to collect venue fields. (Define `showDirections` as `format !== "VIRTUAL" && hasMappableVenue`, then add tests for HYBRID with venue present, HYBRID with venue missing, IN_PERSON with venue missing, and VIRTUAL.)
- The plan says the workflow picker stays active-only, but the current picker calls `/api/survey-templates` and `listSurveyTemplates()` returns all templates, so inactive templates are already selectable and the new Active semantics remain confusing. (Add an explicit active-only picker endpoint/query and a controlled `includeId=<pinned inactive id>` path for editing existing steps.)
- Editing a workflow step that pins a now-inactive template is unspecified; filtering the picker to active-only without rendering the inactive current selection can make the UI appear blank or invite accidental re-picks, while silently preserving it is also unclear. (Render the inactive pinned template as a disabled/current option labeled inactive, preserve it unless the admin changes it, and add a concise picker hint explaining “Active controls auto-attach/default choices; pinned workflow steps keep using their selected template.”)
- Wave 11-G still conflicts with the plan’s Not touched section: git history shows workflow files were committed as recently as 2026-05-11 and there are no current local diffs, but history alone does not prove the parallel-instance coordination lock has lifted. (Make Wave 11-G explicitly blocked/cut to ENH-MAY12-2 unless the user confirms the lock is lifted before implementation; do not count it in the current deploy estimate or verification end-state until then.)

## Low
- The test list misses several RED-GREEN cases specific to the new scope: override-enabled directions template behavior, absence of literal conditional markers, helper-level inactive pinned lookup, active-only auto-match regression for registration and POST_EVENT auto-attach, picker active filtering, inactive pinned edit rendering, and Trigger Now parity. (Add these tests to PLAN.md so the implementation cannot pass by only covering the happy paths already listed.)
