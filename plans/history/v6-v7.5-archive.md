#### Task 3 — Seed `src/prisma/seed-rockefeller-assessment.ts`

**Path correction per Codex H2**: lives at `src/prisma/seed-rockefeller-assessment.ts`, NOT at repo root.

Verbatim content from the plan's Rockefeller seed data section. **NOT upsert-idempotent** (Codex M3 — silent mutation of published versions corrupts historical comparability). Instead:

**All branches below execute inside `db.$transaction(async (tx) => { ... })` interactive transaction (Codex Round 2 M1) AND begin with a transaction-scoped advisory lock (Codex Round 3 M3) so a crash between template insert and version insert cannot leave a half-baked state, AND concurrent prod seeds serialize through the lock — not through unique-constraint races.**

**Codex Round 3 M3 — advisory lock as the first statement inside the transaction:**

```sql
SELECT pg_advisory_xact_lock(hashtext('assessment-rockefeller-v1-seed'));
```

Lock is released automatically at transaction commit/rollback (`_xact_` variant). Two operators running the seed simultaneously will block on the lock instead of both observing State A and racing into a unique-constraint failure.

1. Inside the transaction (after acquiring the advisory lock), look up `AssessmentTemplate` by `alias = "RockHabits"` AND its child `AssessmentTemplateVersion(versionNumber=1, language="enUS")` in one read.
2. **State A — NOTHING found**: create the template (metadata only — `name = "Rockefeller Habits Checklist"`, no `aggregationMode` at this level, no email copy at this level) AND create version 1 with full content (10 sections, 40 questions, scoringConfig, `aggregationMode = "FULL_VISIBILITY"` (Rockefeller does NOT use CEO_ONLY per v7.5), `invitationSubject` + `invitationBodyMarkdown` placeholder copy modeled on Verne preface tone, **`publishedAt = new Date()` + `publishedBy = "system-seed"` (Codex Round 3 H1 — engages the immutability trigger; without this, the trigger is a no-op on the seeded row)**). Compute `contentHash` on the version row before insert.
3. **State B — template + version 1 found, `contentHash` matches expected snapshot**: no-op + log `"already seeded, snapshot matched, no-op"` and exit 0.
4. **State C — template + version 1 found, `contentHash` DIFFERS from expected**: ERROR + non-zero exit + log the contentHash diff. Operator must publish a new versionNumber=2 via Prisma Studio (or, post-v1.5, the admin authoring UI) rather than mutating version 1. The DB trigger already blocks the mutation attempt at the database level, so this branch is mostly a defensive log.
5. **State D — template exists but version 1 missing** (half-baked prior seed that crashed between inserts): self-heal — create the missing version 1 row inside the same transaction with the same `publishedAt`/`publishedBy` setup as State A, log `"healed half-baked seed state, version 1 inserted"` and exit 0.
6. **State E — version 1 exists but template missing**: impossible under Prisma FK, but log + exit non-zero for manual investigation if it ever shows up.

```ts
function scoreSubmission(version: AssessmentTemplateVersion, answers: Answer[]): ScoreResult
```

Returns: `perQuestion[]`, `perSection[]`, `overallTotal`, `overallAverage`, `countAchieved`, `tier`, `tierMetricValue`, `unansweredKeys`.

Test file `src/__tests__/lib/assessments/scoring.test.ts`. Write RED first. Required tests:

1. **Rockefeller golden fixture** (the captured one in the plan): answers Q1_1..Q10_4 → expect `countAchieved=37`, `overallTotal=85`, `overallAverage=2.125`, `tier.label="Great"`, `tier.message="That is a great overall score."`
2. **Tier-gap rejection (Codex Round 1 M4 + Round 2 M4 fix)**: Zod refine reads `scoringConfig.tierMetric` and computes the metric's valid domain from the template's scored questions:
   - `countAchieved` → `[0, questionCount]` (Rockefeller: `[0, 40]`).
   - `overallTotal` → `[sum(question.scale.min), sum(question.scale.max)]`. Rockefeller has all `min=0` so the lower bound is 0, but the formula is `sum(min)` — Codex Round 2 caught the hardcoded `0` lower bound (breaks templates with non-zero minimum scales). For Rockefeller: `[0, 120]`.
   - `overallAvg` → `[scale.min, scale.max]` ONLY if every question in the version shares the same scale. **Codex Round 2 M4**: if scales differ across questions (mixed-scale template), Zod refine rejects `tierMetric: 'overallAvg'` outright with a typed error — averaging across mixed scales is meaningless. Rockefeller's all-0-to-3 case is uniform so `overallAvg` would be valid (Rockefeller actually uses `countAchieved`, so this is forward-compat).
   Refine rejects `scoringConfig` whose `tiers[]` has gaps or overlaps within the **derived** domain. One test fixture per metric type (countAchieved, overallTotal with non-zero min, overallAvg uniform-scale, overallAvg mixed-scale rejection) to cover the derivation logic.
3. **v6.6 strict validation rejections (Codex M5 fix)** — `scoreSubmission()` throws a typed `ScoringValidationError` with a `code` field. **NOT an HTTP status** — HTTP mapping happens in future route handlers, not in this pure function. One test per code:
   - `code: 'UNKNOWN_STABLE_KEY'` — unknown stableKey in answers
   - `code: 'OUT_OF_RANGE'` — value outside `[scale.min, scale.max]` (e.g., 5 when max=3)
   - `code: 'MISSING_REQUIRED_KEY'` — Rockefeller has 40 required; one missing → throws
   - `code: 'EMPTY_ANSWERS'` — empty answers object → throws (all 40 required missing)
   - `code: 'NON_INTEGER'` — fractional value (2.5) via `Number.isInteger` check
   - `code: 'INVALID_TYPE'` — string value (`"2"`), `NaN`, `null`, `undefined`
   - **`code: 'DUPLICATE_STABLE_KEY'` (Codex Round 2 M3)** — answers payload contains the same `stableKey` twice. Without this guard, an adversarial payload submits two values for the same required question and gets order-dependent scoring. Test: `[{ stableKey: 'Q1_1', value: 0 }, { stableKey: 'Q1_1', value: 3 }]` → throws.
4. **Edge cases**: all max (countAchieved=40, Great tier); tie at tier boundary — `count=33` → Great, `count=32` → OK (boundary inclusivity per plan's tier definition).

Iterate red → green → refactor. Land green.

> *Dynamic logic engine — CONTAINED SHAPE INTO v1 for INVITED templates*
>
> Jeff verbatim: *"I'd like to be able to build new assessments without having custom development for that assessment."* Scaling Up Assessment has two distinct logic patterns:
> 1. **Conditional report copy**: paragraph text in the report changes based on a participant's answer. E.g. employee count Q answered 50–149 → render paragraph A; answered 8–24 → render paragraph B.
> 2. **Peer-benchmark comparison**: report panel shows "Your peers averaged 5.2; you're at 3.6" — hardcoded benchmark numbers in v1; real-data benchmarks in v2 once enough submissions accrue.
>
> v7.4's "hardcode Website Assessment + SLIDER_LIKERT only" rescope is preserved for the PUBLIC path (Website Assessment + SunHub results renderers stay hardcoded). For **INVITED templates** v7.5 adds a contained logic-engine shape:
> - `AssessmentTemplateVersion.reportConfig Json?` gains two optional fields:
>   - `conditionalSections: [{ stableKey, sortOrder, when: { stableKey, op, value }, markdownContent }]`. Each section evaluates `when` against the participant's `answers` + `submission.result`. If true, the section's `markdownContent` renders in the report. `op` enum (v1): `'eq' | 'between' | 'gt' | 'lt' | 'in'`. `value` shape varies by `op` (`between` takes `[low, high]`; `in` takes a string array; others take a number or string). **No AND/OR composition in v1** — one stableKey reference per section. If Jeff needs composition later, v2 adds it.
>   - `peerBenchmarks: { [stableKey: string]: number }`. Hardcoded benchmark values for v1. Report renderer pulls these to draw the peer-comparison panel. Missing benchmark for a stableKey → panel hides for that question, doesn't break.
> - Report renderer (`<AssessmentResultView>`) evaluates `conditionalSections` server-side at render time and inlines the matching markdown via the existing markdown renderer.
> - Scaling Up Assessment seed ships with placeholder `conditionalSections` matrix + placeholder benchmarks. **Admin UI for editing `conditionalSections` and `peerBenchmarks` is deferred to v1.5** (per Codex co-validate: the when-clause builder is the highest-complexity Wave 2 screen and Scaling Up Assessment's content matrix isn't ready anyway — Jeff is still authoring tier messages for scores 7 & 10). Until the v1.5 authoring UI ships, admins update via direct DB edit / Prisma Studio. The renderer-side evaluation (`<AssessmentResultView>` reading the seeded JSON) still ships in v1 — it's a small evaluator, not a UI.
> - Jeff is flexible on placement (questionnaire vs report renderer) — v1 puts the logic in the renderer, not the survey form.

## Architectural decisions

1. **Recursive `OrgTeam` hierarchy** (`parentTeamId` self-FK, unbounded depth, no CHECK constraint on depth). Adjacency-list patterns; one recursive CTE for tree traversal via Prisma `$queryRaw`. Simpler than `Division+Team` and matches Esperto's actual model.
2. **Immutable `AssessmentTemplateVersion`** with monotonic `versionNumber` per `(templateId, language)`. Editing a template publishes a new version; campaigns pin a specific `versionId`. Each question carries a stable `key` (e.g., `Q1_1`) that survives across versions for longitudinal comparability.
3. **`scoringConfig` JSON validated by Zod**, with explicit `tierMetric` driving tier resolution. Pure scoring function in `lib/assessments/scoring.ts`. Frozen at submit.
4. **Attributable responses** (per user decision). Leader / admin sees per-respondent answers by name.
5. **Magic-link respondent identity**, durable `dedupeKey` is the longitudinal anchor. `OrgRespondent.dedupeKey` is required: `externalId` if present, else `normalize(email)` (lowercased, trimmed). Soft delete preserves history. Admin merge tool re-points submissions to the surviving record.
6. **`OrganizationMembership` enforced from v1**. ADMIN/STAFF auto-membership via seed; non-admin access requires explicit grant. `canAccessOrganization()` is the chokepoint.
7. **Sync invite send, manual resend**. No Inngest queue. Tokens: 32 random bytes, base64url, sha256-stored, `crypto.timingSafeEqual` compare. Revoke = delete the invitation row.
8. **Single canonical compute path**: `submission.answers` is truth. `submission.result` is a frozen cache (computed once at submit, never recomputed). `TEAM_AGGREGATE` reports are computed on-read, persisted as `AssessmentReport` rows only when an admin saves them.

### Scoring algorithm (definitive, captured)
For each respondent submission (formal contract for `scoreSubmission()`):
```
per_question_total[Q]  = answer (0..3)
per_section_total[S]   = sum of 4 question scores in section S    // 0..12
per_section_avg[S]     = per_section_total[S] / 4                  // 0..3
overall_total          = sum of all 40 question scores             // 0..120
overall_avg            = overall_total / 40                        // 0..3
count_achieved         = count of questions where score >= passThreshold(2)  // 0..40
```
- "Tick" on the habits-checklist page: `per_section_avg[S] >= 1.5` per item (treating section avg as the bar)
- "Pass" / "Achieved": `per_question_score >= passThreshold` (Rockefeller: `>= 2`) drives `count_achieved`
- Tier resolution: applied to the metric named in `scoringConfig.tierMetric` (Rockefeller = `countAchieved`)
